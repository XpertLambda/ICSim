ContextContexte

Why Vehicle Security MattersPourquoi la Sécurité Automobile

In 2015, security researchers Charlie Miller and Chris Valasek demonstrated a remote compromise of a Jeep Cherokee via its cellular interface, gaining control over the CAN bus and ultimately the vehicle's brakes and steering at highway speed. The attack was possible because the CAN protocol - designed in the 1980s for isolated networks - has no authentication, no encryption, and no concept of a trusted sender. En 2015, les chercheurs en sécurité Charlie Miller et Chris Valasek ont démontré une compromission à distance d'une Jeep Cherokee via son interface cellulaire, prenant le contrôle du bus CAN et finalement des freins et de la direction du véhicule à grande vitesse. L'attaque était possible parce que le protocole CAN - conçu dans les années 1980 pour des réseaux isolés - n'a ni authentification, ni chiffrement, ni notion d'émetteur de confiance.

Real-world consequenceConséquence réelle

Any process that can write to the CAN bus has the same authority as a legitimate ECU. There is no way for a receiving module to distinguish an injected frame from a real one. Tout processus capable d'écrire sur le bus CAN a la même autorité qu'un véritable calculateur (ECU). Il est impossible pour un module récepteur de distinguer une trame injectée d'une trame réelle.

Real VehiclesVéhicules Réels

CAN in Production VehiclesLe CAN dans les Véhicules de Production

CAN (Controller Area Network, ISO 11898) was designed by Bosch in 1983 as a cost-effective way to replace hundreds of point-to-point wires with a single two-wire bus. Today every road vehicle sold in Europe and North America contains at least one CAN bus, often several running at different speeds for different systems. Le CAN (Controller Area Network, ISO 11898) a été conçu par Bosch en 1983 comme moyen économique de remplacer des centaines de fils point à point par un unique bus à deux fils. Aujourd'hui, tout véhicule vendu en Europe et en Amérique du Nord contient au moins un bus CAN, souvent plusieurs fonctionnant à des vitesses différentes pour différents systèmes.

Core Properties - and Why They Are DangerousPropriétés fondamentales - et pourquoi elles sont dangereuses

No source addressing. A CAN frame carries an arbitration ID (the signal type) and data. There is no sender field. A receiving ECU cannot know which node sent a frame. Pas d'adressage source. Une trame CAN porte un ID d'arbitrage (le type de signal) et des données. Il n'y a pas de champ émetteur. Un ECU récepteur ne peut pas savoir quel nœud a envoyé une trame.

No authentication. Any node can transmit any arbitration ID. There is no cryptographic signature, no session token, and no challenge-response mechanism. Pas d'authentification. N'importe quel nœud peut transmettre n'importe quel ID d'arbitrage. Il n'y a pas de signature cryptographique, pas de jeton de session, et pas de mécanisme défi-réponse.

Priority by ID. Lower arbitration ID wins bus access during simultaneous transmission. This is a feature for safety (brakes win over audio), but an attacker can exploit it by flooding with a low ID. Priorité par ID. L'ID d'arbitrage le plus bas remporte l'accès au bus lors d'une transmission simultanée. C'est une fonctionnalité de sécurité (les freins l'emportent sur l'audio), mais un attaquant peut l'exploiter en inondant avec un ID faible.

Speed Signal - 0x244Signal Vitesse - 0x244

Big-Endian 16-bit IntegerEntier 16 bits Big-Endian

One byte (8 bits) holds values 0–255. ICSim stores speed as km/h × 100, so 96.56 km/h becomes 9656 - which requires a 16-bit integer (range 0–65535). La valeur est répartie en big-endian sur les octets [3] et [4] : l'octet le plus significatif est placé en premier.

OverviewVue d'ensemble

What is UDS?Qu'est-ce que UDS ?

UDS (Unified Diagnostic Services, ISO 14229) is a diagnostic communication protocol implemented in every modern vehicle ECU. It provides a standardized way for external tools (scanners, reprogramming stations) to query faults, read live data, update firmware, and access protected functions. In this lab, UDS requests are sent directly over CAN - no special hardware required. UDS (Unified Diagnostic Services, ISO 14229) est un protocole de communication de diagnostic implémenté dans tout ECU de véhicule moderne. Il fournit un moyen standardisé pour les outils externes (scanners, stations de reprogrammation) d'interroger les défauts, lire des données en temps réel, mettre à jour le firmware et accéder aux fonctions protégées. Dans ce lab, les requêtes UDS sont envoyées directement via CAN - pas de matériel spécial requis.

Read ECU DataLire les Données ECU

Request the VIN, software version, calibration data, or any Data Identifier (DID) stored in the ECU's memory. Demander le VIN, la version logicielle, les données de calibration, ou n'importe quel identifiant de données (DID) stocké dans la mémoire de l'ECU.

Security AccessAccès Sécurisé

A seed-key challenge-response mechanism that locks protected services (reprogramming, hidden routines) behind a cryptographic gate. Un mécanisme défi-réponse graine-clé qui verrouille les services protégés (reprogrammation, routines cachées) derrière une porte cryptographique.

Execute RoutinesExécuter des Routines

Trigger internal ECU functions such as actuator tests, calibrations, or undocumented manufacturer commands via Routine Control (SID 0x31). Déclencher des fonctions ECU internes telles que des tests d'actionneurs, des calibrations ou des commandes fabricant non documentées via Routine Control (SID 0x31).

The UDS Security Access service (SID 0x27) is a two-step seed-key challenge-response that gates privileged operations such as reprogramming. The ECU generates a fresh 2-byte random seed on every request; you must transform it into the correct key and send it back. A wrong key returns NRC 0x35 (invalidKey). Repeated failures trigger a lockout - NRC 0x36 (exceedNumberOfAttempts) - and the ECU stops accepting key attempts until it is reset. Le service UDS Security Access (SID 0x27) est un challenge-réponse graine-clé en deux étapes qui protège les opérations privilégiées telles que la reprogrammation. L'ECU génère une nouvelle graine aléatoire de 2 octets à chaque requête ; vous devez la transformer en clé correcte et la renvoyer. Une mauvaise clé renvoie NRC 0x35 (invalidKey). Des échecs répétés déclenchent un verrouillage - NRC 0x36 (exceedNumberOfAttempts) - et l'ECU cesse d'accepter les tentatives de clé jusqu'à réinitialisation.

Session prerequisite. The firmware only processes Security Access in the Secret/Programming session (0x03). Attempting from any other session returns NRC 0x7F (serviceNotSupportedInSession). Open session 0x03 before requesting the seed, and keep it alive with TesterPresent - the session times out after 3.5 seconds of silence. Prérequis de session. Le firmware ne traite les requêtes Security Access qu'en session Secret/Programming (0x03). Toute tentative depuis une autre session renvoie NRC 0x7F (serviceNotSupportedInSession). Ouvrez la session 0x03 avant de demander la graine, et maintenez-la avec TesterPresent - la session expire après 3,5 secondes de silence.

The key derivation algorithm - barbhack_vin50. The firmware XOR-s each seed byte with one of the last two ASCII characters of the VIN (the same VIN you found in challenge 04). L'algorithme de dérivation de clé - barbhack_vin50. Le firmware XOR chaque octet de la graine avec l'un des deux derniers caractères ASCII du VIN (le même VIN trouvé au défi 04). Comme le XOR est sa propre inverse, la même formule permet aussi de retrouver la graine depuis une clé connue.

Find the hidden routineTrouver la routine cachée
Hard

Why manual brute-force fails. With three unknown bytes (sub-function XX, routine ID YY, activation ZZ), a naive cansend loop has no way to pair requests with responses. Frames from a previous iteration bleed into the next candump read, and the session times out mid-scan without a synchronized keepalive. You need proper tooling. Pourquoi le brute-force manuel échoue. Avec trois octets inconnus (sous-fonction XX, ID de routine YY, activation ZZ), une boucle cansend naïve ne peut pas associer les requêtes aux réponses. Les trames d'une itération précédente contaminent la lecture candump suivante, et la session expire en plein scan sans keepalive synchronisé. Il faut un outillage approprié.

The approach: a two-pass ISO-TP scanner. L'approche : un scanner ISO-TP en deux passes.

Step 1 - Open an ISO-TP kernel socket. Unlike raw cansend, a SOCK_DGRAM / CAN_ISOTP socket gives you synchronous send() / recv() semantics: every recv() returns exactly the ECU response to the last send(). This eliminates the frame-bleeding problem entirely. Étape 1 - Ouvrir un socket ISO-TP noyau. Contrairement au cansend brut, un socket SOCK_DGRAM / CAN_ISOTP offre une sémantique send() / recv() synchrone : chaque recv() renvoie exactement la réponse ECU au dernier send(). Cela élimine complètement le problème de contamination de trames.

Step 2 - TesterPresent with suppressed response. The keepalive thread sends 0x3E 0x80 every 900 ms. The 0x80 flag (suppressPositiveResponse) tells the ECU to process the frame but send no reply - so the 0x7E 0x00 acknowledgement never arrives on the socket and cannot pollute a subsequent recv(). Why the seed matters. The ECU calls rand() to generate both seed bytes - a new pair every time you request one. You cannot pre-compute the key offline. The algorithm is fixed, but the input changes: you must read the seed from the bus, compute the XOR, and reply - all within the same session window. Pourquoi la graine est importante. L'ECU appelle rand() pour générer les deux octets de graine - une nouvelle paire à chaque demande. Vous ne pouvez pas pré-calculer la clé hors ligne. L'algorithme est fixe, mais l'entrée change : vous devez lire la graine sur le bus, calculer le XOR, et répondre - le tout dans la même fenêtre de session.

Step 1 - Open the Secret/Programming session (0x03). The firmware explicitly checks diagSession == 0x03 before processing any Security Access request. Default (0x01) and Extended (0x02) sessions return NRC 0x7F (serviceNotSupportedInSession). Send 02 10 03 and confirm the 50 03 positive response. Étape 1 - Ouvrir la session Secret/Programming (0x03). Le firmware vérifie explicitement diagSession == 0x03 avant de traiter toute requête Security Access. Les sessions Default (0x01) et Extended (0x02) renvoient NRC 0x7F (serviceNotSupportedInSession). Envoyez 02 10 03 et confirmez la réponse positive 50 03.

Step 2 - Request the seed (02 27 01). The ECU responds with 04 67 01 S0 S1 where S0 and S1 are freshly randomised. Read these two bytes from the response - they are your input to the key derivation function. Étape 2 - Demander la graine (02 27 01). L'ECU répond avec 04 67 01 S0 S1 où S0 et S1 sont fraîchement aléatoirisés. Lisez ces deux octets dans la réponse - ils sont votre entrée dans la fonction de dérivation de clé.
